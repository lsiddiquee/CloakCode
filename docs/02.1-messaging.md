## Command injection — the actuator question (§3.1)

Using VS Code's command execution (`run_vscode_command`):

| Command                             | Result                                                          |
| ----------------------------------- | --------------------------------------------------------------- |
| `workbench.action.chat.open "text"` | ✅ ungated, but **prefill/queue only** (see below)              |
| `workbench.action.chat.newChat`     | ⚠️ exists but **"preconditions not met"** (`when`-clause gated) |
| `workbench.action.chat.submit`      | ❌ "Failed to find command"                                     |
| `workbench.action.chat.acceptInput` | ❌ "Failed to find command"                                     |

**The key twist (proven):** when injected **while the agent loop was busy**, the prompt was
**queued** and **auto-submitted when the loop finished** — arriving ~4 minutes later as a
genuine `user.message` with **no user action**. So the "submit" gap is filled by the
**runtime queue**, not a submit command.

VS Code chat exposes **three send modes** (from the input send-button):

- **Stop and Send** — interrupt the running loop and send now.
- **Add to Queue** (Alt+Enter) — after the current loop (what happened above).
- **Steer with Message** (Enter) — inject **into** the running loop to redirect it.

→ **"Steer with Message" is the ideal remote blocker-answer / redirect primitive.** Its
command ID is client-side and was not confirmable from the server (later resolved — see the M3c
section below).

> **Confound warning:** `queue`/`steer`/`interrupt` keyword hits in the logs are mostly
> this investigation's own meta-conversation content + system-prompt text, **not**
> structural events. There is **no** distinct `queued`/`steer` event type; a dispatched
> queued message looks like an ordinary `user.message`.

**Verdict:** command injection is a _viable but non-deterministic_ actuator (stage/queue
works; precise submit/steer control unconfirmed here — confirmed later in M3c). Owning the loop
stays the robust path for token-live control.

- **§4.2** "Command injection is prefill-only / can't submit" → **INCOMPLETE.** A busy-loop
  injection is **queued and auto-submitted** by the runtime.

## Deterministic per-session send (§4.12)

Verified 2026-07-10. Our observed `sessionId` **is** the local chat-session id — Copilot builds
`vscode-chat-session://local/${base64url(sessionId)}` (`toolCalling.tsx`), matching core's
`LocalChatSessionUri`. So a **two-step** `vscode.open(<that URI>)` →
`workbench.action.chat.open({query})` delivers a remote message to a **specific** session —
proven live. `vscode.open` hosts it in an **editor** (which broke the transcript — moot now: the
debug-log source stays complete for editor sessions, so targeting + observability coexist).
**Panel** targeting is **impossible for extensions**: the capability is the internal
`IChatWidgetService.openSession(uri, ChatViewPaneTarget)`; no extension-invokable command exists
(`_chat.voice.switchToSession` is a controller method; `workbench.action.chat.openInSidebar`
takes no session arg). `session.respond` was generalized (`toolCallId` optional) so it doubles as
a **free-form chat message** channel.

**⚠ Window-local caveat (2026-07-11).** This targeting resolves only a session that lives in the
**window running the command** (the bridge's window). A `sessionId` owned by a **different
window** — or any window with **no bridge** — has no editor/session to open, so `vscode.open` →
`chat.open` **creates a NEW chat** in the bridge's window instead of driving the intended one.
Because `scanSessions` lists **every** on-disk session across all `workspaceStorage/<hash>` (all
stamped with the single bridge `instanceId`) and there is no on-disk owner signal, foreign
sessions are **listed but not actuatable**. Tracked in docs/05 (_cross-window actuation_).

## Steer / queue / stop / stop-and-send — the verified command sequences (M3c)

§3.1 found the **three send modes** (Stop-and-Send, Add-to-Queue, Steer-with-Message) but noted
their command IDs were "client-side and not confirmable from the server." **That thread is now
closed (SHIPPED 2026-07-15):** the driving commands are public `workbench.action.chat.*`, invoked
from the ext host after focusing the session URI (§4.12), and each was **live-confirmed** against a
real running turn. The mapping (also tabulated in docs/03 "Mid-turn flag"):

| Action           | RPC                       | Command sequence (after `vscode.open(<session URI>)`)             |
| ---------------- | ------------------------- | ----------------------------------------------------------------- |
| **Queue / send** | `session.respond {text}`  | `chat.open {query}` — auto-queues while a turn runs, else sends    |
| **Steer**        | `session.steer {text}`    | `chat.open {query, isPartialQuery:true}` → `steerWithMessage`      |
| **Stop & send**  | `session.stop {text}`     | `chat.cancel` → `chat.open {query}`                                |
| **Stop**         | `session.stop {}`         | `chat.cancel` (no-arg)                                             |

Verified mechanics:

- **`isPartialQuery` is the steer/send switch.** Verified in the copilot-chat source and live:
  `chat.open {query}` **alone submits** the message; adding **`isPartialQuery: true`** only
  **populates** the composer without sending — which is exactly what `steerWithMessage` then needs
  (prefill the running turn's next-boundary redirect, don't start a fresh turn).
- **Queue needs no separate command.** Because `chat.open {query}` **auto-queues** when a turn is in
  flight and **sends** when idle (the runtime queue, §3.1/§4.2), one command covers both "queue" and
  "send" — and doubles as the stale-`inTurn` escape hatch (docs/03 "Mid-turn flag"): it does the
  right thing whether or not the turn has actually ended.
- **`chat.cancel` is no-arg and acts on the focused session.** So stop/stop-and-send focus the
  session URI first, then cancel; stop-and-send follows with `chat.open {query}` to send the
  replacement prompt as a fresh turn.
- **None of these leaves an on-disk marker.** A steered/queued message is recorded as a plain
  `user.message` and a cancelled/stopped turn as an ordinary `assistant.turn_end` — no
  `steer`/`queue`/`aborted` tag either way (LIVE-CONFIRMED 2026-07-15). The observer therefore tracks
  only **in-flight-ness** (`inTurn`), never the action type — full detail in
  [02.2 §4.28](02.2-turn-tracking.md).
- **All are `remote-operator` actions** (docs/04), command-only, and **window-local /
  focus-dependent** (§4.12): they drive the session only in the window that owns it.
