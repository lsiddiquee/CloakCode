# 02.5 — Session state, title, telemetry, controls, attachments, forks

> Topic detail for [02 — Research findings](02-research-findings.md). Outcomes live in the main
> doc's ledger; this file keeps the mechanics + wrong turns. Establishes _what lives in VS Code's
> client-side `ChatModel` vs on the server_, and everything derived from it: the session title,
> per-turn telemetry, the model/effort/context/mode controls, attachments, and forks.

---

## VS Code's authoritative `ChatModel` is client-side, not on the server (§4.11)

REFINED (2026-07-10). Only **Copilot's** logs (transcript + debug-log) are server-side. VS Code's
authoritative **`ChatModel`** — the full conversation, the session **title**, per-turn
tokens/credits — is **workbench (client) state**, persisted by `ChatSessionStore` **on window
shutdown** to the **client** machine. Verified on a devcontainer→WSL2→Windows 11 stack: an
exhaustive `~/.vscode-server` scan finds **no** `chatSessions/` / `state.vscdb`; the store lives on
**Windows** at `%APPDATA%\Code\User\workspaceStorage\<hash>\chatSessions\<sessionId>.json` (a
`chatSessionOperationLog` JSONL — `{"kind":0,v:<snapshot>}` + appended `{"kind":1|2,k,v}` patches;
append-incremental, compacted on close). The Windows FS isn't mounted in the container — the remote
boundary is enforced. _Implication: CloakCode (server-side) can only read Copilot's two logs; the
client store is unreachable (and that's aligned with zero-code-sync)._

## Session title is LLM-generated (§4.13)

Verified 2026-07-10. The title VS Code shows is generated by the copilot `ChatTitleProvider` (an LLM
child session) and stored as `ChatModel.customTitle` (client-side). Server-side it's reachable
**only** via the debug-log: `main.jsonl` has a `child_session_ref` (`label:"title"`) →
`title-<childId>.jsonl` whose `agent_response` text **equals** the title (`0ed16d08` → "New chat
session testing" = the Windows store's `customTitle`). `scanSessions` now prefers it
(`debugLogTitle`), falling back to the first user message when there's no debug-log.

## Debug-log telemetry (§4.14)

Catalogued 2026-07-10, **build deferred post-MVP**. Each `llm_request` span carries what the
transcript never had: `model`, agent `debugName`, `inputTokens`/`outputTokens`/`cachedTokens`,
`ttft`, request `__dur`, `maxTokens`, and **`copilotUsageNanoAiu`** (billing); the Windows store
confirms per-turn `copilotCredits`. `tool_call` spans add duration + error; `agent_response` adds
`reasoning`. Surfacing this as session telemetry is a future slice (docs/05).

## A forked conversation gets no transcript of its own — invisible as a distinct row (§4.27)

Confirmed 2026-07-16. Forking a chat in the UI does **not** create a new
`transcripts/<forkId>.jsonl`; the fork's turns land in an **existing** transcript (the source's),
while the fork gets a **new** `debug-logs/<forkId>/` dir. Verified on-disk: session
`7e695673-…-1883bf14b05a` (hash `bf235c2e…`) kept growing while the fork produced turns, and the
fork's own message text landed **inside** `7e695673….jsonl`; the fork's session id
`c62241fb-…-979b26941a75` has a `debug-logs/c62241fb…/` dir but **no** `transcripts/c62241fb….jsonl`.
Because `scanSessions` lists `transcripts/*.jsonl`, it sees the fork folded into the source row, not
as its own — **platform behaviour, not a scanner bug**. **Confirmed (2026-07-16):** re-checked a
**full day** after the fork was created — still **no** `transcripts/c62241fb….jsonl`, so the fork
transcript is **never** created (not a lazy flush). The **detailed mechanism** (why a fork's
transcript id and debug-log id diverge) and the **chosen surfacing approach** are **to be
reviewed/finalized later** — tracked as **post-MVP** scope (docs/05, _Surface forked conversations as
distinct sessions_).

## Attachments: file/text content is inlined & recoverable; image bytes are NOT persisted (§4.29)

Confirmed 2026-07-16. Attachments are modelled **client-side** as `IChatRequestVariableEntry` (core
`chat/common/attachments/chatVariableEntries.ts`): `kind:'file'`, **`kind:'image'`** (`isPasted:true`
for a pasted screenshot; `mimeType`), `kind:'paste'` (pasted **code/text**, not screenshots), …; the
base entry's `value` carries the payload (image **bytes** for images). This is the client-authoritative
`ChatModel` (§4.11), not the server. On disk: a **file/text** attachment's **content is inlined** into
the message (`user.message.content` in the transcript; the `llm_request.attrs.userRequest` request
text in the debug-log) as an `<attachment …>…</attachment>` block → **recoverable** (verified with
strings that only exist inside the attached file and were never typed as commands). But **image /
pasted-screenshot bytes are absent**: a pollution-proof scan for a **≥5000-char** base64 run after a
PNG (`iVBORw0KGgo`) / JPEG (`/9j/4AAQ`) signature found **none** in the transcript or debug-log, a
real **~15 KB** attached image was **not** present, and no `*.png`/`*.jpg` blob is written under
`workspaceStorage/<hash>`. The transcript's `user.message.attachments` array **exists but is always
written empty (`[]`)** — no structured attachment metadata. **Methodology caveat (bit me twice):**
grepping the logs for a marker **pollutes** them — your own `run_in_terminal` arguments are logged, so
`grep "iVBORw0KGgo"` "finds" the signature; use **never-typed** strings + a **long-run** requirement.
⇒ image bytes live only in the client `IImageVariableEntry.value` and stream to the model API;
mirroring them needs a **renderer-side** read (same renderer-reachability question as the actuator).
Recorded as a **known limitation we cannot handle** from the on-disk observer (docs/05, _Known
issues_).

## Session controls (model / thinking-effort / context-size / mode): READ on-disk, SET is arg-driven & session-targeted (§4.30)

Corrected 2026-07-16, deep **core** VS Code source dive — an earlier "picker-only / actuator-only"
call was WRONG. The four chat controls we want to mirror and drive:

- **Options catalog — fully on disk.** `debug-logs/<sessionId>/models.json` is one array (**39
  models** in the sampled session), each with `id`/`name`/`vendor`/`capabilities.family`,
  `model_picker_enabled`, `model_picker_category` (`powerful`/`versatile`/`lightweight`),
  `model_picker_price_category` (the cost dots), `capabilities.limits.max_context_window_tokens`
  (context ceiling, e.g. 1M), `capabilities.supports.reasoning_effort` (per-model effort enum, e.g.
  `low,medium,high,xhigh,max`), and `billing.token_prices.{default,long_context}`. "**Auto** (10%
  discount)" is a **virtual** picker entry computed client-side (`AutoModelDiscountLabel`), not a
  models.json row.
- **Detect (read / emulate) — YES, per-request, from the debug-log** `llm_request.attrs`: `.model`;
  thinking effort in `.requestOptions` → `output_config.effort`; `.requestOptions.thinking`; mode
  from `.debugName` (`panel/editAgent` = Agent; Ask/Plan strings not yet captured locally);
  context-**tier** inferred from `.inputTokens` vs the model's `default.context_max` /
  `long_context.context_max` (there is **no** literal `contextSize` field — long-context is a
  boolean tier toggle, `chat.preferLongContext.enabled` / `applyContextSizeOverride`). Lags one turn
  and the _pending_ selection between turns lives only in the client store
  ([02.3 §4.6](02.3-tool-call-handling.md)/§4.11). A renderer-side read also exists
  (`IChatWidget.input.selectedLanguageModel` / `currentModeObs`;
  `ChatModelConfigurationStore.getModelConfiguration(modelId)`) for an in-window surface.
- **Set (WRITE) — CORRECTED: value-specific, session-targeted commands/APIs EXIST** (core VS Code
  `src/vs/workbench/contrib/chat`, not the copilot-extension layer the first pass wrongly read). The
  same `Action2.run(accessor, ...args)` + `getWidgetBySessionResource(uri)` pattern that made
  steer/stop work: **mode** — `workbench.action.chat.toggleAgentMode {modeId, sessionResource}`
  resolves the session's widget and calls `input.setChatMode(modeId)` (set-by-**value**, only cycles
  when `modeId` is omitted); **model** — `workbench.action.chat.changeModel {vendor,id,family}`
  sets by value (window-wide), and per-session via `input.switchModelByIdentifier(id)` +
  `getWidgetBySessionResource`; **effort + context** — no standalone command, but
  `ChatModelConfigurationStore.setModelConfiguration(modelId, {<schemaKey>: enumValue})` writes them
  (keys/enums come from the model's own `configurationSchema`). **Real caveats (NOT "impossible"):**
  (a) window-local session targeting (a widget exists only where the session is hosted —
  [02.1 §4.12](02.1-messaging.md)), and (b) renderer-vs-ext-host reachability — **live-verify**
  before relying on it (steer/stop from the ext host is strong precedent, but confirm). Sourced from
  core `browser/actions/chatExecuteActions.ts`, `browser/widget/input/chatInputPart.ts`,
  `browser/widget/input/chatModelConfigurationStore.ts`, `common/languageModels.ts`, `browser/chat.ts`.
  **Lesson (again):** read the actual `Action2.run(...)` argument handling before declaring a command
  "picker-only." Slicing + open live-confirmations tracked in docs/05 "Remote session controls".
