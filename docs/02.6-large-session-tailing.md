## The cliff: `readFile(вЂ¦, "utf8")` throws on a >512 MiB debug-log (В§4.31)

Found **2026-07-21** on a real session. `SessionFollower.pump()` loads the tail source **whole** вЂ”
`content = await fs.readFile(this.filePath, "utf8")` вЂ” then parses the entire string
(`packages/extension/src/session-observer.ts`). The preferred tail source is the **debug-log**
(`debug-logs/<id>/main.jsonl`, В§4.10), which for a long agentic session dwarfs the transcript.

Live numbers (session `7e695673`, workspace hash `bf235c2eвЂ¦`):

| file                              | size                                     |
| --------------------------------- | ---------------------------------------- |
| `transcripts/7e695673вЂ¦.jsonl`     | 24.8 MB (reads fine)                     |
| `debug-logs/7e695673вЂ¦/main.jsonl` | **581,573,539 B (~581 MB)**, still growing |

V8 caps a single string at `buffer.constants.MAX_STRING_LENGTH = 536,870,888` chars (~512 MiB).
581 MB exceeds that, so `fs.readFile(debugLog, "utf8")` **throws `ERR_STRING_TOO_LONG`**
(`Cannot create a string longer than 0x1fffffe8 characters`) вЂ” reproduced in ~502 ms with a one-off
`readFileSync`. The log crossed the cap mid-use, so a session that "opened fast yesterday" now never
loads. **Not a code regression** вЂ” no commit touched the parse path; the _data_ crossed a latent
limit (the review's "large stores may degrade" note, now a hard wall).

### Two symptoms that look like a bug but aren't (yet)

1. **Blank content, but tool-call cards still appear.** `session.subscribe` wires **two independent
   followers** (`packages/extension/src/bridge.ts`): a `SessionFollower` for the conversation (from
   the 581 MB debug-log вЂ” throws) and a **separate** `SpoolFollower` for the live-pending overlay
   (from `~/.cloakcode/spool`, a handful of ~1 KB files вЂ” works). So the operator sees pending tool
   cards but no transcript вЂ” a textbook split-source failure.
2. **Nothing in our logs.** `pump()`'s read was wrapped in a bare `catch { return; }` ("transient
   read error") and `SessionFollower` had no logger, so a **permanent** `ERR_STRING_TOO_LONG` was
   indistinguishable from a transient miss and **never logged** вЂ” the client just showed a blank
   session. `findSessionLog`'s transcript read had a second silent `catch {}`. **FIXED (2026-07-21,
   logging half):** the follower takes a `sessionId`-bound child logger and logs every read/turn/watch
   failure **deduped** (a persistent failure logs ONCE, with the file `bytes` вЂ” not every 400 ms poll);
   `findSessionLog` logs a non-ENOENT transcript failure; the bridge logs `rpc.failed` / `rpc.invalid`
   (code + op + traceId, never the message). Codes come from a shared `errorCode` (Node errno / Error
   name вЂ” never the message, which can carry paths/data). **Client surface SHIPPED (2026-07-21):** the
   follower emits a terminal `kind:"error"` subscribe frame (redaction-safe `code` + `bytes`) that the
   web client renders as a reason ("This session is too large to load (N MB)вЂ¦") instead of a silent
   blank.

## Why the observer re-reads the whole file every change (В§4.32)

The follower keeps exactly one piece of state вЂ” `this.emitted`, an **event count** вЂ” and re-reads +
re-parses the **entire** file on every `fs.watch` fire and every 400 ms poll, emitting only
`events[emittedвЂ¦]`. It dedups at the **emit** step, not the **read** step. This is deliberate, and it
is _not_ trivially replaceable, because the parsers (`parseSessionEvents` / `parseDebugLogEvents`)
are **whole-file pure functions** whose new lines reach back and mutate earlier ones:

- **Retroactive resolves.** A tail `tool.execution_complete {toolCallId:X}` emits a `resolve` /
  `updateStatus` against a part `append`-ed near the head; the mapping lives in an in-call
  `interactiveIds: Map<toolCallId, partId[]>`. Parse a tail slice alone and that map is empty в†’ the
  wrong branch runs and the interactive card never clears.
- **Positional ids + `seq`.** Ids are counters (`user-${userIdx++}`, `msg-${msgIdx}`) and every
  `seq = events.length`. A fresh parse of a slice restarts them at 0 в†’ duplicate ids / colliding
  seqs в†’ client part-map corruption.

So re-parsing from byte 0 is what keeps `seq`, the `*Idx` ids, the interactive map, and the
transcriptв†”debug-log `stitchEvents` boundary globally consistent. Correctness-through-simplicity вЂ”
fine at KBвЂ“MB, **O(n) per append / O(nВІ) live**, and it hits the В§4.31 wall at ~512 MiB.

## The fix: offset tail + a resumable JSONL state machine (В§4.32, proposed вЂ” unbuilt)

The logs are **JSONL** (one complete JSON value per line), so the parse itself never changes вЂ”
`JSON.parse(line)` stays. What changes is: stop holding the whole file as a string, and carry the
parser state across reads.

**Framing (the byte layer).** Track a byte `offset`. On change: `fs.stat`; if `size < offset` в†’
truncation/rotation в†’ **reset offset to 0 and reset the state machine**, full reparse; else stream
`[offset, size)` through `readline` over `createReadStream(file, { start: offset })`. Advance `offset`
only to the **last `\n`** consumed; an unterminated trailing line is left unread and re-picked on the
next flush. **The newline is the sole frame delimiter вЂ” JSON validity is only ever tested on a
complete (newline-terminated) line**; a terminated-but-malformed line is skipped _and_ advanced
(today's behaviour). Decode via the stream's `StringDecoder` so a multibyte UTF-8 char split across a
read boundary isn't corrupted (never `toString("utf8")` an arbitrary byte slice).

**Carried state (the parse layer).** Hoist what is currently in-call onto a persistent object:

```ts
{ seq, userIdx, msgIdx, usageIdx, interactiveIds: Map<string, string[]>, stitchDone: boolean }
```

Feed it complete lines; emit `append` / `updateStatus` / `resolve` as derived. Non-interactive tool
completions need **no** map entry (their part id is a pure function of `toolCallId`); only interactive
ones do. Turn-tracking (`computeInTurn` in `pumpTurn`) also reads the transcript **whole** today вЂ” the
same latent cap вЂ” so it must **tail the last N KB** rather than whole-read; `inTurn` only needs the
last `turn_start` / `turn_end`.

**Payoffs.** (a) Kills the В§4.31 cap on the **initial** open too (stream from offset 0 вЂ” never one
big string). (b) Idle polls go 581 MB в†’ `stat` + delta (near-free), keeping the watch + poll
belt-and-suspenders for the overlay-FS missed-inotify case. (c) Same event shapes, same `SessionPart`
contract вЂ” a pure `@cloakcode/extension`-layer change, no protocol churn.

**Not exotic JSON.** A streaming/SAX parser is only needed for a _single_ giant JSON document; JSONL
is line-framed by construction, which is exactly why event logs are written this way. The only new
code is standard `tail -f` plumbing (offset + partial-line buffer + truncation reset).

## Initial load must stream too вЂ” the stitch + the anchor (В§4.33)

The initial `session.subscribe` read hits the **same В§4.31 cap**: today `findSessionLog` does
`parseSessionEvents(await fs.readFile(transcript, "utf8"))` for the history prefix and the follower
then reads the debug-log whole. Both must **stream**, or we only move the wall from tailing to open.

- **Tailing** вЂ” always the debug-log, offset-incremental (above).
- **Initial load** вЂ” decide the transcriptв†”debug-log seam without holding either file:
  1. Peek the debug-log opening (stream until the first K `user_message` spans) в†’ K candidate texts.
  2. **Pass 1 (find):** stream the transcript collecting only a **compact index** of its
     user-message texts + byte offsets (a tiny fraction of the file), then pick the boundary by the
     **longest aligned prefix** of the debug-log's opening sequence вЂ” today's `alignBoundary` logic вЂ”
     generalised to try any of the first K debug texts as the anchor (reorder-robust). Aligning the
     _sequence_ (never a single message) is what disambiguates a **repeated short opening**.
  3. **Pass 2 (emit):** stream transcript `[0, boundaryOffset)` as `tx-` history, then stream the
     whole debug-log as `dl-` lead; one resumable parser carries `seq` across the seam.
  Two streaming passes stay memory-bounded вЂ” you cannot single-pass-emit, because a no-match would
  already have emitted the whole transcript before the fallback could fire. **Fallback on no match
  (open decision):** debug-log alone + **log it** (recent turns intact, older history dropped), _or_
  transcript + debug-log with a possible duplicate seam (no shared id to dedup on) вЂ” greedy
  K-matching makes this rare either way.

### The anchor is user-message TEXT, not tool-call id (verified 2026-07-21)

Which field joins the two files? Empirically, on a real session:

- **User-message text matches exactly.** debug-log `user_message` span `attrs.content` ===
  transcript `user.message.data.content` (e.g. `"i want to inflate a chat sessio nwithвЂ¦"`) вЂ” a
  top-level field in both. **This is the anchor** (today's `alignBoundary` / `dlTexts` key).
- **`toolCallId` does NOT reliably join.** The transcript id (`toolu_вЂ¦__vscode-1783963651675`) shares
  only its **base** (`toolu_вЂ¦`) with the debug-log, and that base appears **only embedded in the
  `llm_request.attrs.inputMessages` blob** (a serialized conversation string, repeated across turns,
  suffix stripped) вЂ” never as a structured field. The debug-log's own `tool_call` spans key on OTel
  **`spanId`** (`000000000000001b`), which shares nothing with the transcript. A tool id can
  **corroborate** but must never be the sole join вЂ” **we cannot rely 100 % on it.**

Conclusion: **stitch on user-message text (greedy, first K); tool-call id is an unreliable secondary
signal only.**

### Repeated short openings (`continue`, `yes`, `go`)

A single message text is ambiguous вЂ” a session with many `continue` turns has that text at many
transcript positions вЂ” so we **never match one message**. We align the debug-log's opening _sequence_
against the transcript by **longest consecutive agreement** (`alignBoundary`): each candidate
`continue` is scored by how far the turns that FOLLOW it keep matching, and those turns always
diverge, so the right occurrence wins. The residual degenerate case вЂ” a **recycled** debug-log
reopening on a short message with near-identical following activity вЂ” is **bounded** (a mis-placed
seam only drops or duplicates _history_; the live tail is unaffected, nothing crashes) and
**detectable** (best score 1 across multiple equal candidates в‡’ low confidence в†’ log it, prefer the
safe fallback). **Sequence match ships now**; harden this degenerate case **later** (deferred until a
real misalignment is observed in the wild) by enriching the fingerprint with **tool names** вЂ” which DO
cross-match, unlike ids (`read_file` === `read_file`, transcript `toolName` vs debug-log
`tool_call.name`) вЂ” and the assistant text between user turns. **Timestamps can't help:** the
transcript is rehydrated with replay time (В§4.22), so its `timestamp`s are fiction after a restart.

в†’ Remediation is **unbuilt**; tracked in [docs/05 roadmap](05-roadmap-and-open-questions.md). The
observability fix (log once + `kind:"error"` frame) can and should ship independently of the tail
rewrite.
