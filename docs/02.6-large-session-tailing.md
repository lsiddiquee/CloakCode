# 02.6 — Large-session tailing & the read-scaling cliff

## The cliff: `readFile(…, "utf8")` throws on a >512 MiB debug-log (§4.31)

Found **2026-07-21** on a real session. `SessionFollower.pump()` loads the tail source **whole** —
`content = await fs.readFile(this.filePath, "utf8")` — then parses the entire string
(`packages/extension/src/session-observer.ts`). The preferred tail source is the **debug-log**
(`debug-logs/<id>/main.jsonl`, §4.10), which for a long agentic session dwarfs the transcript.

Live numbers (session `7e695673`, workspace hash `bf235c2e…`):

| file                              | size                                     |
| --------------------------------- | ---------------------------------------- |
| `transcripts/7e695673….jsonl`     | 24.8 MB (reads fine)                     |
| `debug-logs/7e695673…/main.jsonl` | **581,573,539 B (~581 MB)**, still growing |

V8 caps a single string at `buffer.constants.MAX_STRING_LENGTH = 536,870,888` chars (~512 MiB).
581 MB exceeds that, so `fs.readFile(debugLog, "utf8")` **throws `ERR_STRING_TOO_LONG`**
(`Cannot create a string longer than 0x1fffffe8 characters`) — reproduced in ~502 ms with a one-off
`readFileSync`. The log crossed the cap mid-use, so a session that "opened fast yesterday" now never
loads. **Not a code regression** — no commit touched the parse path; the _data_ crossed a latent
limit (the review's "large stores may degrade" note, now a hard wall).

### Two symptoms that look like a bug but aren't (yet)

1. **Blank content, but tool-call cards still appear.** `session.subscribe` wires **two independent
   followers** (`packages/extension/src/bridge.ts`): a `SessionFollower` for the conversation (from
   the 581 MB debug-log — throws) and a **separate** `SpoolFollower` for the live-pending overlay
   (from `~/.cloakcode/spool`, a handful of ~1 KB files — works). So the operator sees pending tool
   cards but no transcript — a textbook split-source failure.
2. **Nothing in our logs.** `pump()`'s read is wrapped in a bare `catch { return; }` ("transient
   read error"), and `SessionFollower` is never even given a logger. A **permanent**
   `ERR_STRING_TOO_LONG` is therefore indistinguishable from a transient miss and is **never logged
   or surfaced** — the client just shows a blank session. `findSessionLog`'s transcript read has a
   second silent `catch {}`. → Fix regardless of the tail rewrite: **log the failure once**
   (rate-limited, via the bridge logger) and push a `kind:"error"` subscribe frame so the client can
   say "session too large to load".

## Why the observer re-reads the whole file every change (§4.32)

The follower keeps exactly one piece of state — `this.emitted`, an **event count** — and re-reads +
re-parses the **entire** file on every `fs.watch` fire and every 400 ms poll, emitting only
`events[emitted…]`. It dedups at the **emit** step, not the **read** step. This is deliberate, and it
is _not_ trivially replaceable, because the parsers (`parseSessionEvents` / `parseDebugLogEvents`)
are **whole-file pure functions** whose new lines reach back and mutate earlier ones:

- **Retroactive resolves.** A tail `tool.execution_complete {toolCallId:X}` emits a `resolve` /
  `updateStatus` against a part `append`-ed near the head; the mapping lives in an in-call
  `interactiveIds: Map<toolCallId, partId[]>`. Parse a tail slice alone and that map is empty → the
  wrong branch runs and the interactive card never clears.
- **Positional ids + `seq`.** Ids are counters (`user-${userIdx++}`, `msg-${msgIdx}`) and every
  `seq = events.length`. A fresh parse of a slice restarts them at 0 → duplicate ids / colliding
  seqs → client part-map corruption.

So re-parsing from byte 0 is what keeps `seq`, the `*Idx` ids, the interactive map, and the
transcript↔debug-log `stitchEvents` boundary globally consistent. Correctness-through-simplicity —
fine at KB–MB, **O(n) per append / O(n²) live**, and it hits the §4.31 wall at ~512 MiB.

## The fix: offset tail + a resumable JSONL state machine (§4.32, proposed — unbuilt)

The logs are **JSONL** (one complete JSON value per line), so the parse itself never changes —
`JSON.parse(line)` stays. What changes is: stop holding the whole file as a string, and carry the
parser state across reads.

**Framing (the byte layer).** Track a byte `offset`. On change: `fs.stat`; if `size < offset` →
truncation/rotation → **reset offset to 0 and reset the state machine**, full reparse; else stream
`[offset, size)` through `readline` over `createReadStream(file, { start: offset })`. Advance `offset`
only to the **last `\n`** consumed; an unterminated trailing line is left unread and re-picked on the
next flush. **The newline is the sole frame delimiter — JSON validity is only ever tested on a
complete (newline-terminated) line**; a terminated-but-malformed line is skipped _and_ advanced
(today's behaviour). Decode via the stream's `StringDecoder` so a multibyte UTF-8 char split across a
read boundary isn't corrupted (never `toString("utf8")` an arbitrary byte slice).

**Carried state (the parse layer).** Hoist what is currently in-call onto a persistent object:

```ts
{ seq, userIdx, msgIdx, usageIdx, interactiveIds: Map<string, string[]>, stitchDone: boolean }
```

Feed it complete lines; emit `append` / `updateStatus` / `resolve` as derived. Non-interactive tool
completions need **no** map entry (their part id is a pure function of `toolCallId`); only interactive
ones do. Turns stay sourced from the (small) transcript via `computeInTurn`, unchanged.

**Payoffs.** (a) Kills the §4.31 cap on the **initial** open too (stream from offset 0 — never one
big string). (b) Idle polls go 581 MB → `stat` + delta (near-free), keeping the watch + poll
belt-and-suspenders for the overlay-FS missed-inotify case. (c) Same event shapes, same `SessionPart`
contract — a pure `@cloakcode/extension`-layer change, no protocol churn.

**Not exotic JSON.** A streaming/SAX parser is only needed for a _single_ giant JSON document; JSONL
is line-framed by construction, which is exactly why event logs are written this way. The only new
code is standard `tail -f` plumbing (offset + partial-line buffer + truncation reset).

→ Remediation is **unbuilt**; tracked in [docs/05 roadmap](05-roadmap-and-open-questions.md). The
observability fix (log once + `kind:"error"` frame) can and should ship independently of the tail
rewrite.
